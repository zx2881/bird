# `normalize_data_text_to_simplified.py` 说明

## 1. 脚本功能

这个脚本的目标很单一：

- 把 `data/` 目录里用于前端展示的 `name`、`summary` 类文本，统一整理成**简体中文**
- 重点处理两种情况：
  - 英文
  - 繁体中文
- 默认只改展示字段，不改这些基础字段：
  - `id`
  - `english_name`
  - `latin_name`
  - 其他不适合作为中文展示名的源字段

默认处理文件和列：

- `data/birds.csv`
  - `name`
  - `summary`
- `data/locations.csv`
  - `name`
  - `summary`
- `data/relations.csv`
  - `subject`
  - `object`
  - `object_summary`

之所以把 `relations.csv` 里的 `subject` 和 `object` 也纳入，是因为它们在当前项目里实际承担“展示名称”的作用。如果这里只改 `birds.csv`、`locations.csv`，不改 `relations.csv`，后续重新构建图谱时很容易出现名字不一致。

---

## 2. 它不会做什么

这个脚本**不会**做下面这些事：

- 不会直接修改 `public/knowledge.json`
- 不会修改 `english_name`、`scientific_name`、`latin_name`
- 不会改 `id`
- 不会自动扩写事实内容
- 不会把状态代码 `VU`、`EN`、`CR` 这类字段硬翻成中文代码名

换句话说，它的定位是：

**只清洗展示文本，不重构数据结构。**

---

## 3. 运行前提

脚本本身使用 Python 编写，依赖很轻，只额外用到了 `requests`。

因为“英文转简体中文”不是 Python 标准库能直接完成的，所以脚本采用的是：

**可配置的 OpenAI 兼容接口**

也就是说，你需要准备一个可用的文本翻译接口，例如：

- OpenAI
- 兼容 OpenAI Chat Completions 协议的代理或网关
- 你自己的兼容服务

可用环境变量：

```bash
TEXT_NORMALIZER_API_BASE
TEXT_NORMALIZER_API_KEY
TEXT_NORMALIZER_MODEL
```

如果你没设置，脚本会默认：

- `api_base = https://api.openai.com/v1`
- `model = gpt-4.1-mini`

---

## 4. 推荐用法

### 4.1 先扫描，不落盘

先看哪些单元格疑似需要处理：

```bash
python scripts/normalize_data_text_to_simplified.py --scan-only
```

这个模式下脚本只会：

- 读取 CSV
- 找出疑似英文或繁体文本
- 输出数量和示例

它不会调用接口，也不会写回文件。

### 4.2 先做一次试运行

```bash
python scripts/normalize_data_text_to_simplified.py --dry-run
```

这个模式会真正调用翻译接口，但不会回写 CSV，适合先验证翻译结果是否符合预期。

### 4.3 确认无误后正式写回

```bash
python scripts/normalize_data_text_to_simplified.py
```

正式模式下，脚本会：

- 创建备份目录
- 写回 CSV
- 保存翻译缓存

### 4.4 如果你怀疑还有繁体漏网

可以启用更强的统一模式：

```bash
python scripts/normalize_data_text_to_simplified.py --translate-cjk-all
```

这个参数的意思是：

**只要是目标列里的中文文本，也全部送去统一成简体中文。**

好处是覆盖更全，代价是调用次数和成本会更高。

## 5. 脚本运作底层逻辑

这一段尽量用容易懂的话来讲。

### 第一步：先读 CSV

脚本会读取：

- `birds.csv`
- `locations.csv`
- `relations.csv`

但它不是把所有列都改一遍，而是只看“和展示文本有关”的列。

### 第二步：先判断“这格内容像不像要处理”

脚本不会一上来就把所有文本都送去翻译接口，因为那样：

- 成本高
- 速度慢
- 也更容易误改已经是正常简体中文的内容

所以它会先做一个粗判断。

它主要看三件事：

1. 这段文本里有没有明显英文单词
2. 这段文本里有没有明显繁体中文特征
3. 这段文本是不是本来就不该翻，比如 `VU`、`EN` 这种状态代码

如果判断为“不像要处理”，就跳过。

### 第三步：给每种文本一个“语义范围”

脚本不会把所有字符串都当成同一种东西来翻。

它会区分：

- 鸟类名称
- 地点名称
- 栖息地名称
- 威胁因素名称
- 摘要文本

这样做的原因很简单：

同一段英文，在不同场景下可能需要不同的中文表达。  
把“鸟名”和“地名”分开缓存，能减少错译和串译。

### 第四步：先查缓存，再决定要不要调用接口

脚本会维护一个缓存文件：

`data/.simplified_translation_cache.json`

逻辑是这样的：

- 这段原文以前翻过，就直接复用
- 没翻过，才真正去调用接口

这样做有三个好处：

- 相同名字不会重复花钱
- 同一个词在多个 CSV 中更容易保持一致
- 反复重跑脚本时会更快

### 第五步：调用翻译接口，但要求它“只返回最终文本”

脚本给模型的要求非常严格：

- `name` 字段只输出名字
- `summary` 字段只输出摘要
- 不要解释
- 不要加引号
- 不要加“翻译如下”

这样做的目的，是为了让模型输出能直接写回 CSV。

### 第六步：写回前先备份

正式写回时，脚本会先把原始文件复制到：

`data/_backup_before_simplify/时间戳/`

这样如果结果不满意，你可以直接回退原文件。

### 第七步：最后才真正写回 CSV

只有在前面的步骤都完成后，脚本才会把新文本写回原文件。

所以它的整体顺序其实就是：

**先判断 -> 再翻译 -> 再备份 -> 最后写回**

这个顺序是为了尽量稳。

---

## 6. 为什么不用“全量无脑翻译”

表面上看，最简单的方法像是：

“把 `name` 和 `summary` 全扔给翻译接口，翻完覆盖回来。”

但在这个仓库里这样做风险比较大，因为：

- `relations.csv` 里有状态代码
- 不同 CSV 之间有名称一致性要求
- 某些字段虽然有英文缩写，但并不应该翻掉
- 有些中文本来就已经是简体，不值得再次改写

所以这个脚本用了“先筛选、再翻译”的方式，而不是无脑全量改写。

---

## 7. 建议的实际操作顺序

建议按这个顺序跑：

1. `--scan-only`
2. `--dry-run`
3. 正式写回
4. `npm run build:data`
5. `npm run build`

这样你可以先确认清洗范围，再确认翻译结果，最后再重建前端数据产物。

---

## 8. 可能要注意的地方

- 如果你的翻译接口风格太“自由”，个别名字可能会被翻得不够稳定
- 如果你发现某些繁体内容没有被扫描出来，可以用 `--translate-cjk-all`
- 如果你想重新翻译已经缓存过的内容，可以加 `--refresh-cache`
- 如果你只是想先看影响范围，不要直接正式运行，先用 `--scan-only`

---

## 9. 一句话总结

这个脚本不是“万能翻译器”，它更像是一个**面向当前仓库结构的展示文本清洗工具**：

- 只改该改的列
- 尽量不碰主键和源字段
- 先扫描
- 有缓存
- 有备份
- 最终目标是把前端展示文本统一成简体中文
